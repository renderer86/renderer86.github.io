# Virtual Shadow Map 개요

<div class="vsm-post">
<p style="color:var(--text2);line-height:1.85;">
UE5의 Virtual Shadow Map(VSM)은 고해상도 Shadow Map을 GPU VRAM 한계 없이 사용하기 위해, OS의 Virtual Memory 개념을 GPU Shadow에 적용한 기술이다. 이 글에서는 기존 Shadow Map의 한계에서 시작해 VSM의 핵심 아이디어, 구조, 그리고 Shadow Cache의 Uncached → Clear → Rebuild 과정을 UE5 소스 코드를 통해 분석한다.
</p>
<span class="section-eyebrow">01 — 배경</span>
</div>

# 배경: CSM의 한계

<div class="vsm-post">
<p style="color:var(--text2);line-height:1.85;">
UE5 이전에 Unreal Engine이 주로 사용한 Shadow 기법은 <strong>Cascaded Shadow Map(CSM)</strong>이다. CSM은 카메라 거리에 따라 Shadow Map을 여러 단계(Cascade)로 나눠, 가까운 거리는 고해상도·먼 거리는 저해상도를 할당하는 방식이다. 현실적인 타협이었지만, UE5 규모의 오픈 월드에서는 여러 한계가 겹쳐 드러났다.
</p>

<div class="card-grid">
<div class="card coral">
<div class="card-label">한계 1</div>
<div class="card-title">먼 거리의 shadow detail 저하</div>
<div class="card-desc">CSM의 각 Cascade는 고정된 세계 영역을 고정 해상도로 커버한다. 카메라에서 멀어질수록 Shadow Map의 1 texel이 담당하는 세계 면적이 커지고, 그림자 경계가 흐려지거나 계단 현상이 생긴다. Cascade 수를 늘려도 근본적으로 해결되지 않는다.</div>
</div>
<div class="card gold">
<div class="card-label">한계 2</div>
<div class="card-title">해상도가 화면 픽셀 수요와 무관하게 고정</div>
<div class="card-desc">CSM은 Cascade 경계로 해상도가 결정된다. 실제로 화면에 몇 픽셀이 그림자를 필요로 하는지와 상관없이, 보이지 않는 영역까지 동일한 해상도로 Shadow Map 전체를 렌더링한다. 해상도를 화면 픽셀 수요에 맞게 유동적으로 조절할 수 없다.</div>
</div>
<div class="card purple">
<div class="card-label">한계 3</div>
<div class="card-title">로컬 라이트 확장성 부재</div>
<div class="card-desc">CSM은 Directional Light에 특화된 구조다. Point/Spot Light가 수십~수백 개인 오픈 월드 씬에서 각각 Shadow Map을 주려면 메모리와 렌더링 비용이 폭발적으로 증가해, 현실적으로 대부분의 로컬 라이트는 그림자를 포기해야 했다.</div>
</div>
<div class="card blue">
<div class="card-label">한계 4</div>
<div class="card-title">정적 씬에서도 매 프레임 재렌더링</div>
<div class="card-desc">CSM은 캐싱 구조가 없다. 아무것도 움직이지 않는 정적 환경에서도 Shadow Map 전체를 매 프레임 다시 렌더링한다. 정적 오브젝트가 대부분인 씬에서 막대한 GPU 시간을 낭비한다.</div>
</div>
</div>

<p style="color:var(--text2);line-height:1.85;">
이 한계들의 공통 원인은 결국 하나다. <strong>해상도를 높이면 메모리와 렌더링 비용이 그대로 따라온다.</strong> 16K Shadow Map으로 detail 문제를 정면 돌파하면 어떨까? Directional Light 1개 기준으로 16개 Clipmap Level × 16K = 약 <strong>20.8 GB</strong>가 필요하다. VRAM에서 현실적으로 쓸 수 없는 수치다. VSM의 "Virtual"은 바로 이 딜레마를 푸는 열쇠다.
</p>

<span class="section-eyebrow">02 — 핵심 아이디어</span>
</div>

# VSM의 핵심 아이디어: Virtual Memory

<div class="vsm-post">
<p style="color:var(--text2);line-height:1.85;">
OS의 Virtual Memory는 매우 큰 가상 주소 공간을 할당하되, 실제로 접근하는 부분만 물리 메모리(Physical Memory)에 올려두는 개념이다. VSM은 이 개념을 GPU Shadow에 그대로 적용한다.
</p>

<p style="font-size:11px;color:var(--text3);margin:8px 0 0;"><code>r.Shadow.Virtual.MaxPhysicalPages</code>로 제한</p>
</div>
</div>

<p style="color:var(--text2);line-height:1.85;">
가상 16K 텍스처를 <strong>128×128 픽셀 단위의 Virtual Page</strong>로 쪼갠다. 그러면 16K 텍스처는 128×128 = 16,384개의 Virtual Page로 구성된다.
</p>

<p style="color:var(--text2);line-height:1.85;">
Depth Pass가 끝나면 현재 카메라 시점의 GBuffer 픽셀을 분석한다. 각 픽셀의 UV와 Depth를 World 좌표로 변환하고, 다시 Shadow Space로 변환하면 해당 픽셀에 영향을 미치는 Virtual Page를 알 수 있다. 이렇게 실제로 <em>사용되는</em> Virtual Page만 모아서 Physical Page Pool에서 공간을 할당한다.
</p>

<div class="callout callout-warn">
<div class="callout-title">Page Pool Overflow — 그림자가 갑자기 사라진다</div>
<p>
장면이 넓고 Depth 분포가 다양하면 설정된 <code>MaxPhysicalPages</code>를 초과할 수 있다. 이 때 초과된 페이지는 Physical Pool에서 공간을 얻지 못하고, 해당 영역의 그림자가 통째로 사라지는 현상이 발생한다. 픽셀 단위의 노이즈가 아니라 page(128×128px) 단위로 그림자 블록이 없어지기 때문에 시각적으로 매우 두드러진다.<br><br>
<code>r.Shadow.Virtual.ShowStats 2</code>에서 <strong>Allocated</strong>가 <code>MaxPhysicalPages</code>에 근접하면 Overflow 직전이다. <strong>Cleared</strong>가 갑자기 급증하는 시점(카메라 컷 전환, 많은 Dynamic 오브젝트 동시 등장)에 Overflow가 터지기 쉽다. 대응 방법은 두 가지다.<br>
① <code>r.Shadow.Virtual.MaxPhysicalPages</code>를 늘려 Physical Pool 크기를 확보한다 (VRAM 추가 사용).<br>
② <code>r.Shadow.Virtual.ResolutionLodBiasDirectional</code>로 page당 커버 범위를 넓혀 필요한 page 수 자체를 줄인다 (그림자 해상도 하락).
</p>
</div>

<span class="section-eyebrow">03 — 왜 빠른가</span>
</div>

# VSM이 빠른 이유 — 필요한 Shadow만 유지한다

<div class="vsm-post">
<p style="color:var(--text2);line-height:1.85;">
CSM이 느린 근본 원인은 <strong>"Shadow Map 전체 = 렌더링 대상 전체"</strong>라는 등식이다. 보이지 않는 영역도, 변하지 않은 정적 배경도, 화면에 그림자가 닿지 않는 멀리 있는 오브젝트도 매 프레임 똑같이 처리된다. VSM은 이 등식을 깨는 것에서 출발한다.
</p>

<div class="callout callout-teal">
<div class="callout-title">핵심 원칙</div>
<p>VSM은 Shadow를 그리는 게 아니라, <strong>필요한 Shadow만 유지한다.</strong> 이번 프레임에 화면 픽셀이 실제로 필요로 하는 shadow page만 Physical Pool에 올리고, 변한 page만 Rebuild하고, 나머지는 이전 프레임 캐시를 그대로 쓴다.</p>
</div>

<p style="color:var(--text2);line-height:1.85;margin-top:20px;">

</p>

</p>

# 구조: Shadow Depth와 Shadow Projection

<div class="vsm-post">
<p style="color:var(--text2);line-height:1.85;">
VSM의 런타임 동작은 크게 두 단계로 나뉜다. <strong>Shadow Depth Pass</strong>는 Physical Page Pool에 Depth를 기록하고, <strong>Shadow Projection</strong>은 Lighting Pass에서 그 결과를 읽어 그림자 계수를 계산한다. 두 단계 사이에 Shadow Cache의 Merge 결과물이 놓이며, Lighting은 항상 이 최종 합성 데이터를 입력으로 받는다.
</p>
</div>

## Shadow Depth Pass

<div class="vsm-post">
<p style="color:var(--text2);line-height:1.85;">
Shadow Depth Pass는 그림자를 드리우는 Geometry를 Physical Page Pool에 렌더링하는 과정이다. 핵심은 각 Geometry가 <em>어떤 Virtual Page에 영향을 미치는지</em> 파악해 해당 Physical Page에만 Depth를 쓰는 것이다. Geometry는 Nanite 메시와 Non-Nanite 메시로 나뉘며 처리 경로가 다르다.
</p>

<div class="card-grid">
<div class="card teal">
<div class="card-label">Nanite Mesh</div>
<div class="card-title">Nanite 래스터라이저 — Cluster → Shadow Page</div>
<div class="card-desc">
Nanite Shadow는 Visibility Buffer와 동일한 철학을 Shadow Depth에 적용한다. 일반 렌더링에서 <em>Cluster → Screen Pixel → VisBuffer → Material Resolve</em>인 것처럼, Shadow에서는 <em>Cluster → Shadow Page → Physical Page에 Depth 기록</em>이다.<br><br>
Cluster는 128 Triangle 묶음이며, GPU가 각 Cluster가 shadow page에 실제로 기여하는지를 직접 판별한다. 작은 Cluster는 <strong>SW Rasterizer</strong>(Compute로 직접 래스터라이즈), 큰 Cluster는 <strong>HW Rasterizer</strong>(고정 함수 파이프라인)로 분기한다. CPU 개입 없이 GPU 안에서 필요한 Cluster만 골라 해당 Physical Page에 Depth를 쓰므로, 수백만 폴리곤 메시도 shadow page 기여분에 비례한 비용만 발생한다.
</div>
</div>
<div class="card blue">
<div class="card-label">Non-Nanite Mesh</div>
<div class="card-title">CullPerPageDrawCommands → ExecuteIndirect</div>
<div class="card-desc">오브젝트별로 어떤 shadow page에 영향을 미치는지 GPU 컴퓨트 셰이더(<code>CullPerPageDrawCommands</code>)로 판별한다. 영향받는 Uncached page가 있는 오브젝트만 모아 <code>ExecuteIndirect</code>로 렌더링한다. 이 판별 과정에서 4단계 Culling이 적용된다.</div>
</div>
</div>
</div>

### Non-Nanite: CullPerPageDrawCommands 4단계 Culling

<div class="vsm-post">
<p style="color:var(--text2);line-height:1.85;">
Non-Nanite 오브젝트는 Nanite의 Cluster 단위 Culling이 없으므로, shadow page에 불필요한 렌더링을 막기 위해 별도로 4단계 Culling을 거친다. 각 단계는 앞 단계를 통과한 오브젝트에만 적용되어, 비용이 큰 판별일수록 뒤에 위치한다.
</p>

<div class="step-block s1">
<h4>1. Frustum Cull</h4>
<p>오브젝트의 Bounding Box를 Shadow View의 절두체(Frustum)와 비교한다. Shadow View 공간 밖에 있으면 어떤 shadow page에도 영향을 줄 수 없으므로 즉시 제거된다. 가장 먼저, 가장 빠르게 걸러낸다.</p>
</div>

<div class="step-block s2">
<h4>2. Empty Rect Cull</h4>
<p>Frustum을 통과한 오브젝트의 Bounding Box를 Shadow Space에 투영해 2D Rect를 구한다. 이 Rect가 아무 Virtual Page도 커버하지 않으면(Empty Rect) 그릴 필요가 없으므로 제거된다. 예를 들어 매우 먼 거리에 있어 shadow space에서 서브픽셀 크기로 투영되는 경우가 해당된다.</p>
</div>

<div class="step-block s2">
<h4>3. Page Mask Cull</h4>
<p>Rect가 page를 커버하더라도, 그 page들이 모두 이미 캐시된(Cached) 상태라면 다시 렌더링할 필요가 없다. <code>OverlapsAnyValidPage</code> 함수가 해당 Rect와 겹치는 page 중 <code>UNCACHED</code> Flag가 설정된 것이 하나라도 있는지 확인한다. 이 때 매 page를 직접 순회하는 대신, 미리 만들어둔 <strong>PageFlag Mipmap</strong>을 사용해 넓은 영역을 한 번에 조회한다.</p>
</div>

<div class="step-block s3">
<h4>4. HZB Cull (Cache Invalidation 시)</h4>
<p>이동하거나 변형되는 오브젝트는 영향받는 shadow page를 UNCACHED로 표시(Invalidation)해야 한다. 그런데 해당 오브젝트가 현재 카메라 시점에서 다른 Geometry에 완전히 가려져(Occluded) 있다면, 그 오브젝트의 그림자는 어차피 보이지 않는다. HZB(Hierarchical Z-Buffer)로 이를 확인해 완전히 가려진 오브젝트의 Invalidation을 생략한다. <code>r.Shadow.Virtual.Cache.InvalidateUseHZB</code> CVar로 제어된다.</p>
</div>

<div class="callout callout-info">
<div class="callout-title">PageFlag Mipmap이 필요한 이유</div>
<p>오브젝트 하나가 커버하는 shadow page 범위가 넓을 수 있다. 모든 page를 개별적으로 UNCACHED 여부를 확인하면 느리다. <code>GenerateHierarchicalPageFlags</code> 패스가 미리 PageFlag Mipmap을 생성해두고, <code>OverlapsAnyValidPage</code>에서 오브젝트 Rect에 맞는 Mip Level을 골라 4 texel Gather로 넓은 범위를 한 번에 조회한다.</p>
</div>

<p style="color:var(--text2);line-height:1.85;margin-top:24px;">
4단계를 모두 통과한 오브젝트는 <code>DrawIndirectArgs</code> 버퍼에 등록되고, <code>RasterPasses</code>의 <code>ExecuteIndirect</code>로 실제 Shadow Depth가 렌더링된다. <strong>Vertex Shader</strong>에서는 Instance별 <code>PageInfo</code>를 읽어 해당 Shadow View로 변환한다.
</p>

<div class="code-block"><div class="code-lang">HLSL — RasterPassVS</div><span class="cm">// InstanceCulling_PageInfoBuffer에서 해당 Instance의 ViewId를 가져온다</span>
PackedPageInfo = InstanceCulling_PageInfoBuffer[InstanceIdIndex];
FPageInfo PageInfo = UnpackPageInfo(PackedPageInfo);

<span class="cm">// PageInfo.ViewId에 해당하는 Shadow View로 월드 좌표를 변환</span>
FNaniteView NaniteView = UnpackNaniteView(
    ShadowDepthPass_PackedNaniteViews[PageInfo.ViewId]);
PointClip = <span class="fn">mul</span>(float4(PointTranslatedWorld, <span class="num">1</span>),
    NaniteView.TranslatedWorldToClip);

<span class="cm">// Clip 좌표를 Physical Page 위치로 Scale/Bias</span>
<span class="fn">ScaleBiasClipToPhysicalSmPage</span>(NaniteView, PointClip, ClipPlanesOut, PageInfo);</div>

<p style="color:var(--text2);line-height:1.85;"><strong>Pixel Shader</strong>에서는 Physical Page 주소를 계산해 Depth를 <code>InterlockedMax</code>로 기록한다.</p>

<div class="code-block"><div class="code-lang">HLSL — RasterPassPS</div><span class="cm">// Virtual Page → Physical Page 주소 변환</span>
FShadowPhysicalPage Page = <span class="fn">ShadowDecodePageTable</span>(
    ShadowDepthPass_VirtualSmPageTable[
        <span class="fn">CalcPageOffset</span>(NaniteView.TargetLayerIndex,
                       NaniteView.TargetMipLevel,
                       vAddress >> <span class="num">7u</span>).GetResourceAddress()]);

<span class="kw">if</span> (Page.bThisLODValidForRendering)
{
    <span class="cm">// Physical Page의 실제 픽셀 주소 계산</span>
    uint2 pAddress = Page.PhysicalAddress * (<span class="num">1u</span> << <span class="num">7u</span>)
                   + (vAddress & ((<span class="num">1u</span> << <span class="num">7u</span>) - <span class="num">1</span>));
    <span class="cm">// Static 페이지는 ArrayIndex=1, Dynamic은 ArrayIndex=0</span>
    <span class="kw">const int</span> ArrayIndex = PageInfo.bStaticPage
        ? <span class="fn">GetVirtualShadowMapStaticArrayIndex</span>() : <span class="num">0</span>;
    <span class="cm">// Depth 기록 (더 가까운 값 유지)</span>
    <span class="fn">InterlockedMax</span>(ShadowDepthPass_OutDepthBufferArray[
        uint3(pAddress, ArrayIndex)], <span class="fn">asuint</span>(DeviceZ));
}</div>

<div class="callout callout-info">
<div class="callout-title">Physical Page Pool 구조</div>
<p><code>Shadow.Virtual.PhysicalPagePool</code>은 <code>R32_UINT</code> 포맷의 2D Array 텍스처다. ArrayIndex 0은 Dynamic(전경) 데이터, ArrayIndex 1은 Static(배경) 캐시 데이터를 저장한다. 이 두 레이어 분리가 Shadow Cache의 핵심이다.</p>
</div>
</div>

## Shadow Projection

<div class="vsm-post">
<p style="color:var(--text2);line-height:1.85;">
Lighting Pass에서 그림자를 계산할 때는 역방향으로 진행한다. GBuffer 픽셀의 World Position을 Shadow Space로 변환 → Virtual Page 주소 계산 → Page Table 참조로 Physical Page 주소 획득 → Physical Page Pool에서 Depth 읽기. Shadow Depth Pass와 대칭적인 Virtual → Physical 변환을 거쳐 저장된 Depth와 현재 픽셀 Depth를 비교해 그림자 계수를 구한다.
</p>

<div class="flow-row">
<div class="flow-step">
<div class="step-num">Step 1</div>
<div class="step-name">World → Shadow Space</div>
<div class="step-desc">GBuffer Pixel을 Shadow View 행렬로 변환</div>
</div>
<div class="flow-arrow">→</div>
<div class="flow-step">
<div class="step-num">Step 2</div>
<div class="step-name">Virtual Page 계산</div>
<div class="step-desc">Shadow Space UV로 Virtual Page Address 결정</div>
</div>
<div class="flow-arrow">→</div>
<div class="flow-step">
<div class="step-num">Step 3</div>
<div class="step-name">Page Table 조회</div>
<div class="step-desc">Virtual → Physical Page 주소 변환</div>
</div>
<div class="flow-arrow">→</div>
<div class="flow-step">
<div class="step-num">Step 4</div>
<div class="step-name">Depth 비교</div>
<div class="step-desc">Physical Page Pool Depth와 비교해 Shadow 계수 출력</div>
</div>
</div>

<span class="section-eyebrow">05 — Shadow Cache</span>
</div>

# Shadow Cache: Uncached → Clear → Rebuild

<div class="vsm-post">
<p style="color:var(--text2);line-height:1.85;">
Shadow Depth를 매 프레임 모든 Page에 대해 새로 렌더링하면 비용이 매우 크다. 정적 오브젝트(Static Mesh)는 이동하지 않으므로 이전 프레임의 Depth를 재사용할 수 있다. 이것이 Shadow Cache의 존재 이유다.
</p>
</div>

## Static / Dynamic 분리

<div class="vsm-post">
<div class="card-grid">
<div class="card teal">
<div class="card-label">Static Cache (ArrayIndex = 1)</div>
<div class="card-title">배경 레이어</div>
<div class="card-desc">이동하지 않는 Static Mesh의 Depth를 저장한다. 씬이 바뀌지 않는 한 매 프레임 재사용된다. Physical Page Pool의 Array Slice 1에 보관된다.</div>
</div>
<div class="card blue">
<div class="card-label">Dynamic Cache (ArrayIndex = 0)</div>
<div class="card-title">전경 레이어</div>
<div class="card-desc">이동하거나 애니메이션 중인 오브젝트의 Depth를 담는다. 매 프레임 Static Cache 위에 덮어써서 최종 결과를 만든다. Array Slice 0에 저장된다.</div>
</div>
</div>

<p style="color:var(--text2);line-height:1.85;">
페이지 상태는 Flag로 관리된다. 두 UNCACHED Flag의 조합으로 해당 페이지가 어떤 처리가 필요한지를 표현한다.
</p>

<div class="flag-row">
<span class="flag-badge flag-coral">VSM_FLAG_DYNAMIC_UNCACHED (bit 1)</span>
<span class="flag-badge flag-gold">VSM_FLAG_STATIC_UNCACHED (bit 2)</span>
<span class="flag-badge flag-blue">VSM_FLAG_ALLOCATED (bit 0)</span>
<span class="flag-badge flag-teal">VSM_EXTENDED_FLAG_VIEW_UNCACHED (bit 6)</span>
</div>

<div class="code-block"><div class="code-lang">HLSL — VirtualShadowMapPageAccessCommon.ush</div><span class="cm">// 페이지가 할당되었는지</span>
<span class="kw">#define</span> VSM_FLAG_ALLOCATED           (<span class="num">1U</span> << <span class="num">0</span>)
<span class="cm">// Dynamic 레이어가 유효하지 않아 재렌더링 필요</span>
<span class="kw">#define</span> VSM_FLAG_DYNAMIC_UNCACHED    (<span class="num">1U</span> << <span class="num">1</span>)
<span class="cm">// Static 레이어가 유효하지 않아 재렌더링 필요</span>
<span class="kw">#define</span> VSM_FLAG_STATIC_UNCACHED     (<span class="num">1U</span> << <span class="num">2</span>)
<span class="cm">// 이 View 자체가 Uncached 상태 (라이트가 이동했거나 첫 프레임)</span>
<span class="kw">#define</span> VSM_EXTENDED_FLAG_VIEW_UNCACHED (<span class="num">1U</span> << <span class="num">6</span>)</div>
</div>

## Uncached 상태가 되는 경우

<div class="vsm-post">
<p style="color:var(--text2);line-height:1.85;">
Uncached 상태는 View 레벨과 Page 레벨 두 단계로 발생한다.
</p>

<div class="step-block s1">
<h4>View Uncached — 라이트가 이동했거나 첫 프레임</h4>
<p>
CPU 측의 <code>FVirtualShadowMapPerLightCacheEntry</code>는 <code>Prev.RenderedFrameNumber</code>로 캐시 유효성을 추적한다. 라이트 방향이 바뀌거나, 강제 Invalidate가 발생하거나, 처음 렌더링되는 경우 <code>Prev.RenderedFrameNumber = -1</code>로 설정된다.
</p>
<p>
이 값이 0 미만이면 <code>bIsUncached = true</code>가 되고, <code>Projection.bUnCached</code>를 통해 GPU 셰이더에 전달된다. GPU에서는 이 페이지에 <code>VSM_EXTENDED_FLAG_VIEW_UNCACHED</code>가 설정되어 모든 페이지를 강제 재렌더링한다.
</p>

<div class="code-block"><div class="code-lang">C++ — VirtualShadowMapCacheManager.cpp</div><span class="cm">// UpdateClipmap: 라이트 방향이 바뀌면 캐시 무효화</span>
<span class="kw">if</span> (bForceInvalidate ||
    LightDirection != ClipmapCacheKey.LightDirection ||
    FirstLevel != ClipmapCacheKey.FirstLevel)
{
    Prev.RenderedFrameNumber = -<span class="num">1</span>; <span class="cm">// 캐시 무효</span>
}

<span class="cm">// RenderedFrameNumber < 0 이면 bIsUncached = true</span>
<span class="kw">bool</span> bNewIsUncached = Prev.RenderedFrameNumber < <span class="num">0</span>;

<span class="cm">// Uncached ↔ Cached 전환 시에도 한 번 더 Invalidate</span>
<span class="cm">// (Static 페이지가 초기화되지 않았을 수 있으므로)</span>
<span class="kw">if</span> (bNewIsUncached != bIsUncached)
{
    Prev.RenderedFrameNumber = -<span class="num">1</span>;
    bIsUncached = bNewIsUncached;
}</div>
</div>

<div class="step-block s1">
<h4>Page Uncached — 새로 할당되거나 오브젝트가 이동한 경우</h4>
<p>
새로 Physical Page를 할당받는 경우(<code>AllocateNewPageMappings</code>), 해당 페이지는 아직 한 번도 렌더링되지 않았으므로 반드시 재렌더링해야 한다. 이 경우 <code>DYNAMIC_UNCACHED | STATIC_UNCACHED</code> 두 Flag가 모두 설정된다.
</p>
<p>
이동하는 오브젝트는 <code>CullPerPageDrawCommands</code>에서 <code>MarkPageDirty</code>를 통해 영향받는 페이지에 UNCACHED Flag를 설정한다. Static Mesh는 <code>STATIC_UNCACHED</code>, Dynamic Mesh는 <code>DYNAMIC_UNCACHED</code>에만 영향을 준다.
</p>

<div class="code-block"><div class="code-lang">HLSL — VirtualShadowMapPhysicalPageManagement.usf</div><span class="cm">// 신규 페이지 할당 시 — 두 레이어 모두 Uncached</span>
uint Flags = VSM_FLAG_ALLOCATED
           | VSM_FLAG_DYNAMIC_UNCACHED
           | VSM_FLAG_STATIC_UNCACHED
           | RequestDetailGeometryFlag;

<span class="cm">// 오브젝트가 이동해서 Static을 무효화한 경우</span>
<span class="kw">if</span> (bStaticInvalidated)
{
    NextPageFlags |= (VSM_FLAG_STATIC_UNCACHED | VSM_FLAG_DYNAMIC_UNCACHED);
}
<span class="cm">// Dynamic만 무효화된 경우 (Dynamic 오브젝트 이동)</span>
<span class="kw">else</span>
{
    NextPageFlags |= VSM_FLAG_DYNAMIC_UNCACHED;
}</div>
</div>

## Clear — Uncached 페이지 초기화

<div class="vsm-post">
<p style="color:var(--text2);line-height:1.85;">
Uncached Flag가 설정된 페이지는 재렌더링 전에 반드시 초기화해야 한다. <code>SelectPagesToInitializeCS</code>가 MaxPhysicalPages개의 스레드로 실행되며 페이지별 처리를 결정한다.
</p>

<div class="code-block"><div class="code-lang">HLSL — SelectPagesToInitializeCS</div>[numthreads(VSM_DEFAULT_CS_GROUP_X, <span class="num">1</span>, <span class="num">1</span>)]

    {
        <span class="cm">// 미사용 페이지, 패스</span>
    }
    <span class="kw">else if</span> (bFullyCached)

</p>

    <span class="kw">else</span>
    {
        <span class="cm">// Static 레이어도 Uncached (신규 페이지 or 라이트 이동) → 0으로 클리어</span>
        OutPhysicalPagePool[BasePos + uint3(<span class="num">0</span>,<span class="num">0</span>,<span class="num">0</span>)] = <span class="num">0U</span>;

<div class="callout callout-teal">
<div class="callout-title">Static Cache 활용의 핵심</div>
<p>Dynamic Uncached이지만 Static이 유효한 경우(<code>DYNAMIC_UNCACHED</code>만 설정, <code>STATIC_UNCACHED</code>는 없음), Dynamic 레이어를 0으로 클리어하는 대신 <strong>Static 캐시 레이어를 복사해서 초기화</strong>한다. 이후 Rebuild에서 Dynamic 오브젝트만 추가로 그리면 되므로, Static Geometry를 매 프레임 재렌더링하는 비용을 완전히 회피한다.</p>
</div>

<div class="callout callout-info">
<div class="callout-title">Stats 연결 — Physical Pages: Cleared</div>
<p><code>r.Shadow.Virtual.ShowStats 2</code>의 <strong>Cleared</strong> 수치는 이 Clear 단계에서 초기화된 page 수다. 정적인 씬에서 카메라만 움직이면 Clipmap이 panning되어 새 page가 소량 할당되므로 Cleared가 낮게 유지된다. 반대로 라이트가 이동하거나 많은 Dynamic 오브젝트가 움직이면 Cleared가 크게 증가한다.</p>
</div>
</div>

## Rebuild — 실제 렌더링

<div class="vsm-post">
<p style="color:var(--text2);line-height:1.85;">
Clear가 완료된 Uncached 페이지에 Shadow Depth를 다시 렌더링한다. <code>CullPerPageDrawCommands</code>에서 이미 각 오브젝트가 어떤 페이지에 영향을 미치는지 판별해 DrawIndirectArgs 버퍼를 채워두었으므로, <code>RasterPasses</code>에서 <code>ExecuteIndirect</code>로 필요한 오브젝트만 그린다.
</p>

<div class="card-grid">
<div class="card blue">
<div class="card-label">DYNAMIC_UNCACHED만 설정</div>
<div class="card-title">Dynamic 오브젝트만 재렌더링</div>
<div class="card-desc">Static Geometry는 이미 Static Cache에 있으므로 스킵한다. 이동하는 캐릭터, WPO 적용 오브젝트 등 Dynamic으로 분류된 것만 다시 그린다.</div>
</div>
<div class="card coral">
<div class="card-label">STATIC_UNCACHED도 설정</div>
<div class="card-title">모든 오브젝트 재렌더링</div>
<div class="card-desc">신규 페이지이거나 라이트가 이동한 경우다. Static + Dynamic 모두 그린다. 완료 후 Static 캐시에도 저장한다.</div>
</div>
</div>

<div class="callout callout-warn">
<div class="callout-title">Static Cache를 항상 유지하는 이유</div>
<p>연속적으로 이동하는 라이트는 매 프레임 <code>bIsUncached = true</code>가 되어 전체 Uncached 경로를 탄다. 코드 주석에서 밝히듯 "<em>continuously moving lights will automatically take the uncached path</em>"가 설계 의도다. 한 프레임이라도 멈추면 Static Cache 구축을 시작해 다음 프레임부터는 Dynamic만 재렌더링하는 효율적인 경로로 전환된다.</p>
</div>

<p style="color:var(--text2);line-height:1.85;">
Rebuild가 완료된 후에는 <code>SelectPagesToMerge</code> → Merge 패스에서 Dynamic 레이어(ArrayIndex 0)와 Static 레이어(ArrayIndex 1)를 합성해 최종 Physical Page Pool을 완성한다.
</p>

<div class="callout callout-teal">
<div class="callout-title">Merge 결과물이 Lighting의 실제 입력이다</div>
<p>
Shadow Projection — 즉 Lighting Pass에서 그림자를 계산할 때 읽는 Depth 데이터는 항상 <strong>Merge된 결과</strong>다. Static 배경만 있는 page는 Static 레이어가 그대로 Merge 결과가 되고, Dynamic 오브젝트가 지나가는 page는 "Static 레이어 위에 Dynamic 레이어를 덮어쓴" 합성 결과가 된다. Lighting은 이 두 레이어가 합성된 단일 depth 값을 받아 그림자 계수를 계산한다. 구조상 Lighting 셰이더는 Cache가 있는지, Static인지 Dynamic인지 신경 쓰지 않아도 된다 — 항상 Merge된 올바른 Depth가 Physical Page Pool에 준비돼 있다.
</p>
</div>

<div class="callout callout-info">
<div class="callout-title">Stats 연결 — Static/Dynamic Cached · Invalidated · Merged</div>
<p>
<strong>Static Cached</strong>: Static 레이어가 유효해 그대로 재사용된 page 수.<br>
<strong>Static Invalidated</strong>: <code>VSM_FLAG_STATIC_UNCACHED</code>가 설정돼 Static 레이어를 새로 렌더링한 page 수. Static 오브젝트가 이동하거나 신규 page가 할당될 때 증가한다.<br>
<strong>Dynamic Invalidated</strong>: <code>VSM_FLAG_DYNAMIC_UNCACHED</code>가 설정돼 Dynamic 레이어를 새로 렌더링한 page 수.<br>
<strong>Dynamic Merged</strong>: Static 레이어는 유효하고 Dynamic만 Uncached인 page — Static을 복사해 초기화한 뒤 Dynamic 오브젝트만 추가로 그린 page 수. 이 수치가 높을수록 Static Cache를 잘 활용하고 있다는 의미다.
</p>
</div>

<div class="callout callout-warn">
<div class="callout-title">Nanite 메시는 Non-Nanite Instances 항목에 나타나지 않는다</div>
<p>Culling 통계의 Non-Nanite Instances 항목은 <code>CullPerPageDrawCommands</code>를 통과하는 오브젝트만 집계한다. Nanite 메시는 Nanite 자체 래스터라이저 경로를 타므로 이 통계에 포함되지 않는다. Nanite Shadow 비용은 별도로 GPU 프로파일러에서 확인해야 한다.</p>
</div>
</div>

<div class="vsm-post">
<div class="callout callout-info">
<div class="callout-title">정리</div>
<p>VSM은 16K Shadow Map의 메모리 문제를 Virtual Memory 개념으로 해결하고, Shadow Cache로 매 프레임 재렌더링 비용을 최소화한다. Page 단위로 Uncached 여부를 추적하고, Clear 단계에서 Static Cache를 재활용하며, Rebuild는 꼭 필요한 오브젝트와 페이지만 대상으로 한다. 이 세 단계가 맞물려 GB 단위 VRAM 없이도 16K급 그림자 품질을 실시간으로 유지할 수 있다.</p>
</div>
